# ADR-006: アクセス許可リスト（継続的認可境界）

## ステータス
承認済み（2026-04-22 改訂：継続的認可境界として位置付けを明文化）

## コンテキスト
不特定多数のGoogleアカウントからのアクセスを制限する必要がある。マルチテナント環境では、テナント単位でアクセス可能なユーザーを事前登録制で管理する仕組みが必要。

ADR-031 で確定したホワイトリスト主義を徹底するため、本ADRを「`(tenantId, email)` の組を唯一の認可境界とし、毎リクエスト参照する継続的認可」に改訂する（2026-04-22）。

## 決定
`tenants/{tenantId}/allowed_emails` コレクションによるアクセス許可リスト方式を採用し、以下を実装レベルの必須条件とする:

1. **唯一の認可境界**: `(tenantId, email)` の組を認可の最終判定に用いる。ドメイン単位・ログインプロバイダ設定側のドメイン縛りは採用しない（ADR-031 既定）
2. **継続的認可**: ログイン時の 1 回限りの判定ではなく、**毎リクエストで allowed_emails を参照**する。キャッシュ依存禁止
3. **既存ユーザー経路でも再チェック**: `firebaseUid` 一致 / email 一致 / `x-user-id` 一致のいずれの経路でも、スーパー管理者以外は allowed_emails 再チェックを実施する
4. **削除即剥奪**: `allowed_email` 削除 = アクセス権剥奪。`user` レコードと進捗データは保持（`allowed_email` 再追加で復旧可能）
5. **保存時正規化**: `.trim().toLowerCase()` を route 層と DataSource 層の両方で適用（防御的多層化）
6. **スーパー管理者は例外**: `super-admin.ts` の判定が真ならば allowed_emails に無関係にアクセス可（break-glass 用途）

## 根拠
- **ゼロトラスト境界**: ID token 有効期限内（最大 1h）でも削除効果を即時反映するため、refresh token revoke（PR #277）と毎リクエスト再チェックを併用する
- **削除・復旧の対称性**: `user` を削除すると進捗データが失われる副作用が重い。`allowed_email` を権限境界、`user` レコードを人物プロファイルとして責務分離することで、復旧時に進捗データを維持できる
- **Codex セカンドオピニオン（2026-04-21）**: 案 B（継続的認可境界）を推奨。`allowed_email` 再追加で復旧可能な点を評価

## 影響
### 実装上の責務
- 管理者がユーザー登録時に `allowed_emails` も同時追加（UI 側で担保）
- 削除時は refresh token revoke とセットで実施（`auth-revoke.ts`、PR #277 で実装済）
- 認証ミドルウェア（`tenant-auth.ts`）は毎リクエスト `isEmailAllowed()` を呼び出す

### 運用上の注意
- 本番 DB の既存ユーザーで allowed_emails 未登録のレコードは、#278 実装前に棚卸し必須（Issue #279 / PR #280 の `scripts/audit-users-vs-allowed-emails.ts` で検出）
- 退職者・離任者は allowed_email 削除で即時アクセス剥奪できる（refresh token revoke と併用で 100% 遮断）
