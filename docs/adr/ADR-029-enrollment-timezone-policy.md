# ADR-029: 受講期限のタイムゾーン基準

## ステータス
Accepted

## コンテキスト

受講期限（`quizAccessUntil`, `videoAccessUntil`）は UTC 日末（`23:59:59.999Z`）で保存される。
管理画面は `toLocaleDateString("ja-JP")` で JST 表示。

JST 環境では UTC 23:59:59.999 = JST 翌日 08:59:59.999 となり、
学習者に有利な方向に約9時間のズレが生じる。

Codex セカンドオピニオン（PR #233 レビュー時）で指摘。

## 検討した選択肢

### A. tenant timezone ベース
- setting に timezone を持ち、deadline を tenant TZ の日末で保存
- 利点: 正確
- 欠点: timezone 管理の複雑さ、DST 対応、全既存データのマイグレーション必要

### B. date-only model
- 日付のみ保存（`2026-06-01`）、判定時に tenant TZ で日末に展開
- 利点: シンプルなストレージ
- 欠点: 判定時にタイムゾーン情報が必要、APIインターフェース変更

### C. 現状維持 + ドキュメント化（採用）
- UTC 基準を明文化、管理画面に「UTC 基準」の注記を追加
- 利点: 変更不要、現行ユーザー（全員 JST）で実害なし
- 欠点: 海外展開時に再検討が必要

## 決定

**選択肢 C: 現状維持 + ドキュメント化** を採用。

理由:
1. 現在のユーザーは全員 JST — ズレは学習者に有利な方向（9時間の猶予増）で実害なし
2. 選択肢 A/B は既存データのマイグレーションと API 変更を伴い、現時点ではコスト対効果が低い
3. 管理画面の表示に UTC 基準の注記を追加することで、管理者の誤認を防止

## 今後の対応

- 海外展開が決まった時点で選択肢 A（tenant timezone）に移行を検討
- その際は `CourseEnrollmentSetting` に `timezone` フィールドを追加し、期限判定を tenant TZ ベースに変更
